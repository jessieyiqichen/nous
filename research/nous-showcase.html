认知层 AI 建模

不只模仿你怎么说话
而是理解你怎么思考

Nous 通过自然对话构建用户的认知模型，并能在没见过的新场景下预测行为——还能检测出你自己都没意识到的认知盲区。

一句话定位：现有 AI 分身在「行为层」模仿表达风格，Nous 在「认知层」建模决策逻辑。我做的不是更像你的分身，是知道分身什么时候会出错的检测器——行为层的质检员。
The Problem

为什么"说话像你"还不够

行为层模仿能快速做到 80 分的"像"，但它只记住了表面模式。换一个没见过的场景，它不知道"你会怎么选"——因为它从没理解过你为什么这么选。

Product · Positioning

跟同类比，Nous 站在哪一层

记忆和 AI 分身是个热门赛道。区别不在「谁记得多」，在「在哪个抽象层次建模用户」——这决定了能不能预测用户没说过的场景。

关于市场，诚实说：我没有精确的市场份额数据，上面是定性的定位对比、不是市场规模测算。记忆 / AI 分身是拥挤赛道，我验证的是一个具体判断——建到「认知层」+ 矛盾检测，是不是比停在行为层更能预测真实的人。
Design · Architecture

三层记忆，对应三种抽象

普通产品的记忆是平的——存对话、检索对话。Nous 是分层的：每一层是对用户的不同抽象层次，向上「归纳压缩」，向下「加载到新对话」。

Core Innovation

矛盾检出：盲区无法自评，只能客观对比

盲区的定义就是"你自己看不见的东西"，所以靠自我评价天然失效。Nous 的解法：做双轨信号分析，把用户"声称的自己"和"实际表现的自己"放在一起比对，不一致处即客观盲区证据。

🗣️ Stated（说的）

用户在对话里声称的自我描述——"我做事很随性""我追求自由"。这是用户想成为或以为自己是的样子。

🎯 Behavioral（做的）

用户在具体场景选择中实际展现的行为——两次选择都偏谨慎求稳。revealed preference，不是 stated。

示例矛盾："你说你随性，但刚才两个选择都很谨慎" → 一条 stated-behavioral 矛盾 = 一次客观盲区证据，也是用户的 Aha Moment（"它说中了我没告诉它、但确实是我的东西"）。
Memory System 视角

另一种看 Nous 的方式：一个 user memory system

前面讲了三层架构 + 矛盾检出。这一节换个视角——Nous 其实就是一个垂直在认知建模场景的 user memory system。这个视角更清楚它解决的工程问题。

什么是 user memory system

一个跨 session 持久化、可检索、能持续校正的用户上下文存储 + 推理系统。Memory 不是数据库——它要能记下来、能找到、能在恰当的时候校正。

Nous 的 Memory 工程实现

🗄️ 分层存储

认知层（9 维 schema）+ 记忆层（101 条离散信号）+ 行为层（surface pattern）——不同抽象层次分别存储，向上归纳压缩，向下加载到新对话。

🔄 跨 session 持久化

用户的认知画像不在单次对话里，而是多次对话累积归纳而成。每次新对话开始，从 memory 加载用户认知模型作为推理上下文。

🔍 Hybrid Retrieval

高频信号用本地 DistilBERT 推理（语义匹配，免费），低频信号 fallback 到 Claude API（深度推断）。API 调用降 60%+——这是 hybrid（语义 + LLM）retrieval 的成本优化。

♻️ 持续校正

"预测 → 验证 → 矛盾反馈"三层闭环 = 背景跑的 memory consolidation。memory 质量不是 ingest 那一刻定的，是不断 distill 出来的。

关键洞察：Memory 不只是存储，是"懂用户的引擎"

普通 storage 只解决"记下来"。一个完整的 memory system 还要解决：记多久、记哪些、怎么检索、何时校正、如何脱敏、谁能访问——这一整套工程才是真正的 memory system。Nous 解决了其中一部分，重点放在"矛盾检出"这件事上——把 memory 的质量控制内化到引擎里。

现阶段 Nous 还没做的几样工程：写入前的自动脱敏（避免存秘密）、可追溯的检索来源（每条结果回到源文件 + 行号）、长会话的上下文压缩。这些是任何成熟 memory system 都需要的事，是这版 Nous 的明确差距。
Engineering

支撑设计的工程决策

每个技术选择都服务于"可验证"这个核心目标——不是为了用技术而用。

结构化输出
用 Claude 的 tool_choice 强制按 JSON Schema 返回认知模型，100% 可解析、可入库。这是整个验证闭环能跑起来的前提，不是锦上添花。
反偏差指令
不假设模型听话，而是假设它有已知偏差（讨好、美化、单一归因），在 prompt 层显式对冲：多因归因、人类化语言、解读深度匹配证据强度。
混合信号提取
高频信号用本地 DistilBERT 推理（免费），低频信号 fallback 到 Claude API，API 调用降 60%+。证明"分类任务该训、开放推断该用 prompt"的边界判断。
被动采集器
常驻后台 agent，从用户日常 AI 对话自动提信号，本地关键词预过滤跳过低信号 session（零 API 成本），认知模型持续进化。
闭环验证
模型生成行为预测 → 用户验证 → 矛盾数据回流修正模型，形成自我改进循环。
★ Eval Methodology

如何评估一个 AI 产品的质量

核心原则一句话：绝不让模型评估它自己。好的 eval 是消除评估者偏差，不是把分数做高。下面这个发现，重塑了我对 AI 产品评估的整个理解。

测试集长什么样

模型从用户认知画像里生成 21 道四选一情境题——比如「周末突然没事，你会 A 在家 / B 朋友聚 / C 一个人出门 / D 工作」。用户答完后，用户的答案就是 ground truth。

然后让 LLM 基于建好的 9 维认知模型预测同样的题（但不让它看到用户的答案）。比对一致性，就是模型质量。

T1 · 偏好预测　用户喜欢什么 T2 · 推理预测　用户在新场景怎么推 T3 · 盲区维度　用户自己看不见的

确定性评分（拿用户真实选择对答案）
LLM 在系统性地给自己打高分。换成确定性评分（选对 1.0 / 选错 0.0，用户本人选择是 ground truth）后，真实水平才暴露。这就是为什么 AI 产品的 eval 绝对不能让模型自评自己。

四条评估设计原则

1

确定性评分，不走 LLM 打分

预测题有标准答案（用户本人作答）。选对 1.0、选错 0.0。消除 LLM 评分的自我迎合偏差。

2

永远对照随机基线

4 选 1 的随机基线是 25%。模型 T1/T2 到 71% = 高出基线 46 个百分点——有了基线，71% 才有意义。

3

不同能力用不同评估方法

盲区维度（T3）连预测都不能自评——自评越修越差（49% → 33%），因为盲区就是看不见的东西。改用客观的矛盾检出评估。

4

主动对冲建模偏差

发现 AI 建模会系统性美化用户，于是加反美化校准指令；发现会过度理想化、忽略执行层弹性，于是在 prompt 里强制追问 stated vs revealed 的 gap。

🔭 设计哲学

模型可以不完美，但必须知道自己在哪里不可靠。Nous 的价值不在"更准的认知模型"，而在"知道行为层什么时候会出错的检测器"。

🧪 评估哲学

好的 eval 是消除评估者偏差，不是把分数做高；不同类型的能力要用不同的评估方法——事实预测用确定性评分，盲区用矛盾检出。
